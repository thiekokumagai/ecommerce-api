# Plano 08-01: Associação de Pedidos a Produtos Reais, Validação e Atualização Dinâmica de Estoque (Backend & Frontend)

Este plano descreve o design técnico e as tarefas de implementação para associar itens de pedido aos produtos e itens de variação (`ProductItem`) reais na base de dados, introduzindo lógica transacional de estoque dinâmico no momento de criação e cancelamento de pedidos.

---

## 🎯 Objetivo

* **Integridade de Dados (Prisma)**: Atualizar o esquema do Prisma para linkar `OrderItem` com `Product` e `ProductItem` (resolvendo a limitação de armazenamento textual puro).
* **Lógica Transacional de Estoque**:
  * Ao criar um pedido, validar se a quantidade solicitada de cada produto ou variação está disponível em estoque.
  * Decrementar dinamicamente o estoque do `ProductItem` correspondente de forma atômica/transacional.
  * Ao cancelar o pedido, reverter a baixa de estoque atômica das variações dos itens correspondentes.
* **Consumo Dinâmico na API**: Substituir quaisquer mockages de estoque nos casos de uso e atualizar o script de sementeiras (`seed.ts`) para gerar pedidos e produtos reais e interligados no PostgreSQL.
* **Componentes Frontend**: Reforçar o sincronismo no frontend para renderizar o estoque atualizado na listagem de produtos imediatamente após novas ações de pedidos ou cancelamento.

---

## 🛠️ Mudanças Propostas

### 1. Banco de Dados (Prisma)

#### [MODIFY] [schema.prisma](file:///c:/sites/podemais/ecommerce-api/prisma/schema.prisma)
Adicionar campos de relação opcionais a `OrderItem` para linkar com `Product` e o `ProductItem` específico (a variação física que guarda o estoque):

```prisma
model OrderItem {
  id              String   @id @default(uuid())
  orderId         String
  productId       String
  productItemId   String?  // Nulo se o produto não tiver variações
  productName     String
  price           Decimal  @db.Decimal(10, 2)
  quantity        Int
  variation       String?  // Ex: "Grape Ice, 3mg"
  imageUrl        String?

  order           Order        @relation(fields: [orderId], references: [id], onDelete: Cascade)
  product         Product      @relation(fields: [productId], references: [id])
  productItem     ProductItem? @relation(fields: [productItemId], references: [id])

  @@map("order_items")
}
```

### 2. Camada de Domínio e Use Cases (Backend)

#### [MODIFY] `domain/entities/order.entity.ts`
* Adicionar os campos `productId` e `productItemId` opcionais à entidade de domínio e sua interface/construtor.

#### [NEW] `domain/use-cases/create-order.use-case.ts`
* Criar caso de uso de criação de pedidos transacional que:
  * Recebe o payload do pedido (dados de cliente, endereço, tipo de pagamento e array de itens contendo `productId` e `productItemId`).
  * Em uma **transação do Prisma ($transaction)**, para cada item:
    * Busca o `ProductItem` correspondente usando o `productId` e/ou `productItemId`.
    * Verifica se `ProductItem.stock` é suficiente para cobrir a `quantity` solicitada. Se insuficiente, lança `BadRequestException` ("Estoque insuficiente para o produto X").
    * Decrementa `ProductItem.stock` pela quantidade comprada.
  * Cria o registro do pedido (`Order` e `OrderItem`) no banco.

#### [MODIFY] `domain/use-cases/cancel-order.use-case.ts`
* Atualizar a lógica do caso de uso de cancelamento para que, caso o pedido seja cancelado com sucesso:
  * Realize um incremento (reversão) de estoque correspondente na tabela `ProductItem` para cada item devolvido, dentro de uma transação isolada para evitar condições de corrida (Race Conditions).
